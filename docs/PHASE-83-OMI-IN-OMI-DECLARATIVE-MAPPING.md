# Phase 83 - OMI-in-OMI Declarative Mapping

Phase 83 adds a canonical `.omilisp` self-map that declares OMI in OMI.

The self-map lives at:

```text
declarations/omi-system.omilisp
```

## Rule

OMI can describe itself declaratively without making the description a new
execution authority.

The self-map is a canonical declaration surface. It is not a runtime rewrite,
not a scheduler, not a graph reducer, and not a replacement for the normative
rule files or constitutional doctrine.

## What The Self-Map Declares

- the causal ladder from canonical bytes to presentation
- byte strata from `0x00..0x1F` through `0x40..0x7F`
- constitutional laws for omicron boundary, CONS aperture, projection,
  receipts, and propagation
- FS/GS/RS/US as segmentation planes
- S-P-O-M as derived triangulation, not source authority
- message-like roles without turning message into an authority class
- major repo surfaces by path

## Authority Boundary

`declarations/omi-system.omilisp` is authoritative as an OMI declaration
surface. It does not supersede:

- `docs/RULES.omi`
- QEMU/kernel witnesses
- `docs/OMI-CONSTITUTIONAL-GEOMETRY.md`

Those sources keep their roles. The self-map names and relates them so the
doctrine can be parsed, projected, and triangulated through the same workbench
machinery as other canonical declarations.

## Receipt Boundary

- `identity_receipt` witnesses the declaration.
- `projection_receipt` witnesses the readout lens.
- `package_receipt` remains a portable serialization witness.
- `view_receipt` witnesses presentation mode.

Lazy, greedy, static, and animated projections of the self-map must preserve
the same identity receipt while changing view/projection receipts.

## Locked Statement

> OMI can describe itself declaratively without making the description a new
> execution authority.
