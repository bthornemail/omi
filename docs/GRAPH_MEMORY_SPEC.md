# Graph Memory Spec

This document defines the first flat memory representation for OMI-GRAPH.

## Status

This is the Phase 1 memory ABI. It is intentionally byte-sized. Wider nodes,
canonical edge serialization, protected regions, and persistent graph images are
future work.

## Memory Model

An OMI graph memory image is a flat byte sequence. It is not currently a block
device, filesystem, or object store.

In Phase 1:

- byte offset is the implicit node address
- byte value determines node class
- adjacent byte pairs determine local adjacency pressure
- BOM ticks transform the view by reversing byte order

The image is generated by `tools/image_builder.c` and normally written to
`vm_image/omi.img`.

## Byte Classes

- `0x00`: NULL node
- `0x01..0x7f`: candidate graph node
- `0x80..0xbf`: control code
- `0xc0..0xff`: hybrid byte-node

## Node Addressing

Node identity is currently positional:

```text
node_id = byte_offset
```

The value at that offset gives the node class. Moving bytes through BOM
inversion changes which value occupies a position, but does not create an
external identity system.

Future SID/OID work may separate observed position, stable semantic identity,
and projection identity.

## Adjacency

Phase 1 adjacency is pairwise and non-overlapping:

```text
(0, 1), (2, 3), (4, 5), ...
```

For an odd-length image, the final byte has no adjacent partner in this pass and
does not contribute to the summary.

## Snapshot Requirements

A valid snapshot must include:

- raw memory bytes
- BOM tick number
- rewrite rule version
- deterministic checksum

The current replay validator computes a checksum for diagnostics. The kernel
currently prints summary counts only.

## Phase 1 CONS Resolution

The first vertical slice evaluates memory as adjacent byte pairs:

- equal adjacent non-zero bytes produce a CONS binding
- any pair containing `0x00` produces NULL collapse
- unequal non-zero pairs produce transient oscillation

This rule is intentionally byte-sized and deterministic so the QEMU kernel and
host replay validator can share one ABI before a wider graph encoding exists.

The normative form of this rule lives in `docs/RULES.omi`. This spec describes
the byte-level ABI; `RULES.omi` describes which graph states are valid,
forbidden, permitted, or optimization targets.

## Deterministic Seed Patterns

`tools/image_builder` supports deterministic image generation. The default
Phase 1 pattern is chosen to produce non-zero counts in all three categories:

```text
0x11 0x11 -> CONS binding
0x00 0x33 -> NULL collapse
0x22 0x22 -> CONS binding
0x44 0x45 -> transient oscillation
```

Repeated across 4096 bytes, this yields:

```text
bindings=1024 null=512 transient=512
```

Other intended patterns:

- `null`: all bytes are `0x00`
- `ramp`: bytes increase modulo 256
- `mixed`: a broader repeated diagnostic pattern

## BOM Inversion

The current BOM operation reverses the byte array in place:

```text
[b0, b1, b2, b3] -> [b3, b2, b1, b0]
```

Applying the operation twice returns the original ordering. This gives Phase 1
a minimal replay invariant:

```text
BOM(BOM(memory)) == memory
```

## Replay Expectations

For the same image and tick count:

- host replay should produce deterministic summaries
- tick `0` and tick `2` should have matching checksums
- tick `1` and tick `3` should have matching checksums
- summary counts should remain stable for the default Phase 1 pattern

## Open Questions

- node width after the first byte-only prototype
- canonical edge serialization
- whether control codes should live in-band or in a protected graph region
- whether future adjacency should be overlapping, indexed, or rule-derived
- whether BOM should permute bytes, reinterpret words, or support multiple
  endian views simultaneously
