* Introduction
This document formalizes the S-P-O (Subject-Predicate-Object) graph as a navigable, executable structure.
Each triple represents a directed edge in the graph, and the collection of all triples forms a
bidirectional knowledge graph that can be traversed in multiple directions.

* S-P-O Triples by Layer

** Layer 1 — Physical (Bits)
- bitstream →[is_transmitted]→ raw_signal
- raw_signal →[decodes_to]→ byte_stream

** Layer 2 — Data Link (Encoding)
- byte →[maps_to]→ gauge_operator
- byte →[belongs]→ codepoint_space

** Layer 3 — Network (Gauge)
- state →[applies]→ operator
- operator →[transforms]→ new_state
- state →[has]→ core_fields

** Layer 4 — Transport (Trace)
- byte_stream →[produces]→ Γ
- Γ →[preserves]→ execution_order

** Layer 5 — Session (OMICRON)
- mode →[controls]→ execution_strategy
- OMICRON →[wraps]→ gauge_step

** Layer 6 — Presentation (Projection π)
- (trace,mode) →[derives]→ META
- π_linear →[projects]→ identity_view
- π_recursive →[projects]→ depth_view
- π_replay →[projects]→ replay_view

** Layer 7 — Application (Semantics)
- trace →[is_interpreted_as]→ structure
- graph →[represents]→ computation
- state →[means]→ meaning

** Layer 8 — Macros (Transform Surface)
- structure →[rewrites_to]→ new_structure
- dot →[expands_to]→ pre-lisp
- macro →[transforms]→ expression

** Layer 9 — Adapters (Boundary Interfaces)
- representation_A →[adapts_to]→ representation_B
- text →[parses_to]→ AST
- AST →[compiles_to]→ omi_ir
- trace →[renders_to]→ org_document
- org →[extracts_to]→ build_artifacts
- omi_triple →[compiles_to]→ datalog_fact
- datalog_rules →[evaluate_to]→ minimal_model
- Y_combinator →[enables]→ recursive_closure
- Z_combinator →[enables]→ guarded_recursion

** Layer 10 — Organization (Structure Surface)
- node →[links_to]→ node
- file →[contains]→ section
- concept →[references]→ concept

* Bidirectional Navigation Rules
For every S-P-O triple (subject, predicate, object), the following navigations are defined:

1. Forward: subject --[predicate]--> object
2. Backward: object <--[predicate_inverse]-- subject
3. Predicate lookup: predicate --> [all_relations]--> (subject, object) pairs

* Executable Graph Queries
The following query patterns are supported:

** Find all objects related to a subject via a predicate:
#+BEGIN_SRC org
subject:byte
predicate:maps_to
#-> gauge_operator
#+END_SRC

** Find all subjects related to an object via a predicate:
#+BEGIN_SRC org
object:gauge_operator
predicate:inverse_of(maps_to)
#-> byte
#+END_SRC

** Find all triples containing a specific predicate:
#+BEGIN_SRC org
predicate:maps_to
#-> (byte, gauge_operator)
#+END_SRC

** Find all triples containing a specific subject:
#+BEGIN_SRC org
subject:byte
#-> [(byte, maps_to, gauge_operator), (byte, belongs, codepoint_space)]
#+END_SRC

* Graph Properties
The S-P-O graph has the following properties:

1. **Layered Structure**: Triples are organized by layer (L1-L10)
2. **Bidirectional**: Every triple can be traversed forward and backward
3. **Typed**: Subjects, predicates, and objects have semantic types
4. **Executable**: The graph can be queried and traversed programmatically
5. **Consistent**: No contradictory triples exist within the same context

* Implementation Notes
This S-P-O graph specification can be implemented as:

1. A finite Datalog fact/rule set for fixed-point logical querying
2. A database table with (subject, predicate, object) columns
3. A graph database with nodes and edges
4. In-memory data structures for runtime traversal

The canonical logic adapter is Datalog-style least fixed-point evaluation, not
ordinary Prolog search. Recursive relations SHOULD be modeled as explicit Y or
Z fixed-point forms so ordering, suspension, and coroutine behavior do not
silently enter the S-P-O graph.

Each triple represents a valid transformation or relationship in the OMI system,
and the graph as a whole represents the complete semantic structure of the
10-layer OSI model.

* Verification
To verify the S-P-O graph is correct:

1. Every triple appears in exactly one layer specification
2. All predicates are defined with clear semantics
3. Bidirectional navigation rules are consistent
4. No layer violations exist (e.g., L1 triple depending on L5 concepts)
5. The graph is acyclic when considering layer dependencies (L1→L2→...→L10)
